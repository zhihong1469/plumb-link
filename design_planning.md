# Plumb-Link 技能体系设计规划文档

> **文档定位**：本文件是 Plumb-Link 技能体系的总设计纲领与最高规范，所有 Agent 对 `.trae` 目录的构建、修改、扩展、重构操作，必须严格遵循本文档约定。

---

## 一、设计目标

基于 Plumb-Link 技能体系，构建一套**分层复用、渐进迭代、稳定可控**的软件开发 AI 协作体系，实现以下目标：

1. **约束标准化**：所有代码输出统一遵循编码规范与平台约束，输出稳定可预期
2. **能力模块化**：专项开发能力封装为独立可插拔技能包，按需加载、跨项目复用
3. **演进自动化**：Agent 可基于本规范自行维护、扩展、优化技能体系，无需人工干预结构设计
4. **安全边界化**：内置多层安全护栏，杜绝危险操作与不符合工程规范的输出

---

## 二、整体架构设计

采用三层平级架构，自下而上分别为：**基础约束层 → 专项能力层 → 输出标准层**，三层职责独立、单向依赖、不可反向嵌套。

| 层级 | 对应目录 | 核心定位 | 生效方式 | 优先级 |
|------|---------|---------|---------|--------|
| L1 基础约束层 | `rules/` | 规定「必须遵守什么、绝对不能做什么」，是所有输出的底线规范 | 项目全局永久生效 | 基础层，所有技能必须遵守 |
| L2 专项能力层 | `skills/` | 封装「某一类任务的完整解决方案」，提供工具、流程、参考资料 | 按需触发加载，对应场景自动激活 | 扩展层，在规则约束内叠加能力 |
| L3 输出标准层 | `templates/` | 规定「输出长什么样」，统一代码骨架与交付格式 | 生成对应类型代码时自动参考 | 标准层，提升输出一致性 |

### 架构原则

- **平级关系**：`rules`、`skills`、`templates` 为同级目录，互不嵌套，均直属 `.trae/` 根目录
- **单向依赖**：技能可以引用规则与模板，规则与模板不得依赖具体技能
- **分层复用**：通用规则下沉，专项能力上浮，避免重复定义
- **渐进扩展**：优先复用现有能力，确有不足再新增文件，不追求大而全

### 目录职责说明

| 路径 | 职责 | 维护规则 |
|------|------|---------|
| `rules/` | 存放所有全局生效的规则文件 | 按数字前缀分层，数字越小越通用，数字越大优先级越高 |
| `skills/` | 存放各独立技能包，每个子目录为一个完整技能 | 每个技能必须包含 `SKILL.md` 作为官方识别入口 |
| `templates/` | 存放通用代码骨架与输出模板 | 仅存放结构模板，不包含业务逻辑 |
| `settings.json` | 项目级全局配置 | 存放上下文范围、沙箱策略、命令白名单等 |
| `design_planning.md` | 本设计文档 | 结构变更必须先更新本文档，再执行修改 |

## 四、核心设计原则

### 1. 分层复用原则

- 通用规范下沉至 `00-` 开头的基础规则，跨项目可直接复用
- 平台专属内容放入 `01-` 层，同平台可复用
- 项目特有内容放入 `02-` 层，仅当前项目生效
- 安全红线独立为最高层级，所有场景强制生效

### 2. 最小化原则

- 一个文件只负责一个主题，禁止编写大而全的单体规则文件
- 一个技能只解决一类问题，职责边界清晰，不做能力溢出
- 能复用现有能力就不新增，避免冗余与不一致

### 3. 渐进式迭代原则

- 技能集不是一次性写完的，随开发过程逐步完善
- 每遇到一次 AI 犯错，就补充一条对应规则
- 每完成一次重复流程，就沉淀为一个技能模板
- 不追求一次性完美，小步快跑、持续优化

### 4. 可执行性原则

- 所有规则必须明确、可落地，禁止「写好代码」「注意质量」等模糊表述
- 所有技能必须包含可执行的步骤或脚本，不能只有概念描述
- 所有安全红线必须有明确的判定标准，可自动检查

## 五、规则文件（rules）编写规范

### 1. 命名规范

- 强制使用 `数字前缀-英文标识.md` 格式
- 数字前缀范围 `00-99`，数字越大优先级越高
- 英文标识使用小写+连字符（kebab-case）
- 示例：`00-skill-generation.md`、`99-security-redline.md`

### 2. 优先级规则

1. 高优先级文件覆盖低优先级文件的同名配置
2. `99-` 开头的安全红线拥有最高优先级，不可被任何其他规则覆盖
3. 子目录内的规则继承父目录优先级，同级文件按文件名排序

### 3. 标准文件结构

每个规则文件必须遵循以下结构：

```markdown
# 规则名称

## 一、适用场景
明确本规则在什么情况下生效

## 二、核心规范
分点列出明确的、可执行的约束，每条只讲一件事

## 三、禁止事项
明确列出绝对不允许的行为

## 四、参考示例（可选）
正确与错误的代码对比
```

## 六、技能包（skills）编写规范

### 1. 技能标准结构

每个技能为一个独立文件夹，必须包含以下内容：

```
skill-name/
├── SKILL.md        # 必需：技能入口与核心指令
├── scripts/        # 可选：可执行脚本（Python/Shell）
├── references/     # 可选：参考文档、手册、数据表
├── agents/         # 可选：Agent元数据配置
├── config.json     # 可选：技能配置文件
└── CONFIG.md       # 可选：配置说明文档
```

### 2. SKILL.md 强制格式

必须包含 YAML 头信息 + Markdown 正文：

```markdown
---
name: skill-name
description: "技能描述，包含触发关键词"
version: "1.0.0"
category: "software|hardware|platform|workflow"
---

# 技能名称

## 适用场景
- 场景1
- 场景2

## 核心能力
1. 能力1
2. 能力2

## 执行流程
1. 步骤1
2. 步骤2

## 依赖工具
- 工具1
- 工具2

## 安全配置
- 安全开关说明
- 用户激活要求
```

### 3. 安全配置要求

所有涉及敏感操作的技能必须包含安全配置：

```json
{
  "security": {
    "enabled": false,
    "require_user_activation": true,
    "activation_required_message": "此技能需要用户手动激活",
    "allowed_operations": ["operation1", "operation2"],
    "blocked_operations": ["dangerous_operation"]
  }
}
```

## 七、全局安全红线

所有规则、技能、Agent 操作必须遵守以下底线，任何情况下不得突破：

1. **禁止危险命令**：不得自动执行 `rm -rf`、`dd`、烧录固件等破坏性操作
2. **禁止越权操作**：不得自动执行 `sudo`、修改系统配置等特权操作
3. **禁止无依据硬件修改**：寄存器配置、设备树修改必须标注依据
4. **禁止静默失败**：所有错误必须明确抛出，不得掩盖报错
5. **禁止脱离规范**：所有输出必须遵循既定编码规范与安全约束

详细内容见 `rules/99-security-redline.md`

## 八、Agent 自我迭代工作流

### 流程1：新增一条规则

1. 判断规则所属层级：通用基础 / 平台专属 / 项目特有 / 安全红线
2. 对应到 `rules/` 下对应前缀的文件
3. 若现有文件主题匹配，则追加到对应章节
4. 若主题不匹配，新建对应前缀的规则文件
5. 保持命名规范与文件结构一致

### 流程2：新增一个技能

1. 确认现有技能无法覆盖需求，确有新增必要
2. 在 `skills/` 下新建技能文件夹，命名遵循规范
3. 创建标准 `SKILL.md` 入口文件，填写完整元数据
4. 按需添加 scripts / references / agents
5. 若有可复用逻辑，优先引用 `skills/shared/` 共享库

### 流程3：优化现有内容

1. 只修改内容，不轻易改动文件结构与命名
2. 修改规则必须保持格式规范统一
3. 修改技能必须更新 `SKILL.md` 描述与能力说明
4. 涉及共享库修改，必须同步检查所有依赖技能

## 十、技能安装到目标项目的规范

### 安装流程

当使用 `scripts/install.py` 将技能安装到目标项目时，必须：

1. **检查目标项目结构**
   - 检查 `.trae` 目录是否存在
   - 检查 `rules/` 目录是否存在
   - 检查 `skills/` 目录是否存在

2. **复制规则文件**
   - 将 `rules/` 目录下的规则文件复制到目标项目
   - 确保安全红线规则（`99-security-redline.md`）始终存在

3. **复制技能文件**
   - 复制指定的技能到目标项目
   - 确保 `SKILL.md` 存在
   - 复制共享工具库

4. **创建项目根 SKILL.md**
   - 在项目根目录创建 `SKILL.md`
   - 说明项目使用的技能和规则
   - 引导大模型正确使用技能

5. **更新注册表**
   - 更新 `.trae/skills/shared/skill_registry.yaml`
   - 记录已安装的技能

### 项目根 SKILL.md 模板

```markdown
---
name: [项目名称]
description: "项目技能使用指南"
version: "1.0.0"
---

# [项目名称] 技能使用指南

## 项目概述
[项目简介]

## 已安装技能

### 软件技能
- `build-linux-app`: Linux应用构建
- `git-commit`: Git代码提交

### 硬件技能
- `gpio-config`: GPIO配置
- `i2c-scan`: I2C扫描

### 平台技能
- [平台技能列表]

## 使用规范

### 1. 技能激活
涉及敏感操作的技能需要用户手动激活：
- 查看 `.trae/skills/[skill-name]/CONFIG.md`
- 修改 `config.json` 中的 `security.enabled = true`

### 2. 安全约束
所有操作必须遵守 `.trae/rules/99-security-redline.md` 中的安全红线

### 3. 技能调用
大模型应优先使用已安装的技能，避免绕过技能直接执行命令

## 技能注册表
查看 `.trae/skills/shared/skill_registry.yaml` 获取完整技能列表

## 配置文件
- `.trae/settings.json`: 项目全局配置
- `.trae/skills/[skill-name]/config.json`: 技能配置
```
